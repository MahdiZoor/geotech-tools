1. Scope & intended use

This toolkit provides three linked modules. A Rockfall Block Size Distribution (RBSD) module derives the size distribution of detaching blocks from the discontinuity network and slope geometry; a 2D slope-profile model gives quick screening along a single representative cross-section; and a 3D DEM-based raster model produces spatial hazard mapping across real terrain. The 2D and 3D modules implement a stochastic, process-based lumped-mass/rigid-body trajectory model — releasing many randomized blocks per source location and aggregating the resulting energy, bounce-height, and passage statistics into design-relevant outputs. The RBSD module feeds the block-size input (and, optionally, the release locations) to those simulators.

2. Rockfall Block Size Distribution (RBSD) module

Most rockfall simulators take the block size distribution as a user input. The RBSD module derives it instead from the rock mass itself, following the standard two-stage logic of the rockfall fragmentation literature: an in-situ block size distribution (IBSD) defined by the joint network, transformed into a rockfall block size distribution (RBSD) actually delivered to the slope after kinematic release and fragmentation. It reuses the surface-slope block-theory engine from a companion rock-block stability tool, run headlessly inside this toolkit.

2.1 Pipeline

The module runs four stages in sequence:

  1. DFN generation. A discrete fracture network is built from the user's joint sets. Each set is sampled by mean orientation (dip / dip direction) with Fisher-distributed poles, a lognormal disc-radius distribution, and a target volumetric fracture-area density P₃₂ (m²/m³). Fractures are placed until P₃₂ is reached within the slope's bounding domain (Dershowitz & Herda 1992).
  2. Block cutting. The DFN is intersected with the slope face, and removable convex blocks are identified using Goodman & Shi (1985) block theory — a block is kinematically removable if it has finite volume and an empty block pyramid. This yields the in-situ block population and their volumes (the IBSD).
  3. Release filter. Each block is classified by kinematic mode (falling, planar sliding, wedge sliding, or stable) and assigned a factor of safety from limit-equilibrium analysis on its governing joint(s). Blocks are "released" when their mode is removable and FS is below a user threshold. This is the IBSD → released-block step that the simple IBSD overestimates if skipped.
  4. Fragmentation. Released blocks are fragmented on detachment to give the deposited RBSD (see 2.2).

The in-situ block sizes can be obtained two ways, selected by the Block size input mode control:

  • DFN-based (the four stages above) — derives the in-situ block size distribution from the joint-set statistics and slope geometry.
  • Mapped distribution — the user enters a measured in-situ block-size distribution directly (assumed lognormal, specified by mean and standard deviation of either block volume or diameter, plus a number of blocks or a total detaching volume). The DFN and kinematic stages are skipped and fragmentation applies directly. This suits sites where block size is characterised from scanline/window mapping, photogrammetry, or a volumetric joint count (Jv) rather than an explicit DFN. Because the kinematic release filter cannot run without geometry, the user states whether the entered distribution represents the already-detaching blocks (fragmented directly) or the full in-situ population (a user-supplied release fraction is then applied first). In this mode the blocks have no coordinates, so the release-location coupling options are unavailable.

2.2 Fragmentation model (RFD)

Fragmentation follows the Rockfall Fragmentation Distribution approach of Ruiz-Carulla, Corominas & Mavrouli (2017). Each released block of volume V either survives intact or breaks, with a volume-dependent survival probability modelled as a logistic function of log-volume:

P_survive(V) = 1 / (1 + exp(−k · (ln V − ln V₅₀)))

where V₅₀ is the volume at which a block has a 50% chance of surviving intact and k sets how sharply survival changes with size. Blocks that break are split into daughter fragments whose volumes follow a truncated fractal (power-law) distribution with exponent set by a fractal dimension b (typically 2.4–2.7), sampled down to a minimum fragment volume and rescaled so total volume is exactly conserved.

Each fragment's volume is converted to an effective diameter for the simulators. By default the conversion uses the "mean of the two largest dimensions" convention (matching the trajectory modules' effective-diameter definition) via the user's aspect ratio; an equivalent-sphere option is also provided.

Verification: total deposited volume equals total released volume to within floating-point precision (~1×10⁻¹³ m³ on test runs), confirming the fragmentation transform conserves mass exactly.

2.3 Release-area geometry

The geometry supplied to this module is the detachment face — the rock slope where blocks come loose — not the travel path. It is defined either parametrically (face dip, dip direction, height, width, number of benches, berm width) or by importing an ASCII OBJ mesh of the face. This is deliberately distinct from the runout profiles used by the 2D and 3D trajectory modules, which operate at a much larger (hundreds-of-metres) scale; mixing the two leads to an unrealistically large fracture domain. Fragmentation is modelled as a single detachment/first-impact event, so the release-area geometry is the only geometry this module needs — the subsequent travel and any path-distributed effects are handled by the trajectory modules.

DFN density cap: the network is capped at 150,000 fractures per joint set. A face-scale slope at realistic joint spacing stays well under this; exceeding it indicates a domain that is too large or a joint density (P₃₂) too high for the chosen disc radius. The tool estimates the required fracture count before running and, if it would exceed the cap, stops immediately with a specific recommendation (a larger disc radius or lower P₃₂, or a smaller release area) rather than attempting an intractable run.

2.4 Coupling to the trajectory modules

The RBSD module exports to the 2D/3D simulators in four ways:

  • ① Fitted distribution. A lognormal is fitted to the fragment diameters and written into the 2D and 3D block-size-distribution fields. Fast; discards each block's release location.
  • ② Discrete size population. The actual fragment population is handed to the simulators, which draw real fragment diameters in sequence (via the forced-diameter hook) instead of sampling the parametric distribution. Preserves the full distribution shape rather than a lognormal fit, but size and launch location remain independent streams.
  • ③ Release locations (optional). The block detachment coordinates are written into the 2D release point and the 3D point-seeder list, so trajectories launch from where blocks actually came loose. Because the release-area coordinate frame does not generally coincide with the simulator's frame, the transferred locations are a starting point the user adjusts to their slope/DEM.
  • ④ Linked population. The faithful coupling: each simulated rock is given a specific fragment's diameter and that same fragment's release coordinate as a matched pair, rather than drawing size and location from independent streams. In 3D, one trajectory is launched per fragment from its own mapped detachment point (bypassing the seeder list); in 2D, each fragment's diameter is paired with a release position obtained by mapping its along-face location into the profile's crest-to-toe range. The diameter sampler and the work-item builder advance the same cursor, keeping the pairing in lock-step. This mode supersedes ② and ③ while active. As with ③, the release-area and simulator coordinate frames differ, so transferred locations are representative placements to be refined; set the 2D run count and 3D trajectory count at least equal to the population size to use every block.
Calibration: the fragmentation parameters (V₅₀, k, fractal dimension, minimum fragment volume) are site- and lithology-dependent. The defaults are illustrative placeholders, not calibrated values; they should be tuned against observed deposit block-size data where available. The IBSD → RBSD step is an area of active research and the output is best treated as a probabilistic envelope rather than a deterministic distribution.

3. Shared physics core

Both modules use the same underlying rebound formulation for a block striking the slope surface. Velocity is decomposed into components normal and tangential to the local surface at the moment of impact. The post-impact normal component follows a velocity-dependent restitution law (so high-speed impacts lose proportionally more energy than slow ones):

Vₙ₂ = −Vₙ₁·Rₙ / (1 + (|Vₙ₁|/9)²)

where Rₙ is the normal coefficient of restitution (material-dependent, Section 7), randomly varied ±10% on every individual rebound to represent natural surface heterogeneity.

The tangential coefficient of restitution (Rₜ) is derived from the slope's surface roughness — a representative obstacle height the block must clear during a rebound — relative to the block's own radius:

Rₜ = 1 / (1 + MOH/R)

where MOH (mean obstacle height) is drawn randomly each rebound from three user-specified roughness classes (70%/20%/10% probability weighting, representing the statistical distribution of obstacle sizes a block actually encounters in the field), and R is the block radius. This is a simplified form of the full published relationship, which also includes a penetration-depth correction term; that term was omitted here after its associated published constants failed to reproduce the source paper's own stated validation example by several orders of magnitude when checked directly — almost certainly a transcription error in the source document rather than a real physical relationship, so rather than guess at corrected constants, the simpler roughness-only form is used and disclosed.

The post-impact tangential velocity couples to the block's rotational state:

Vₜ₂ = √[ R²(I·ω₁² + m·Vₜ₁²)·Rₜ / (I + m·R²) ]
ω₂ = min[ Vₜ₂/R , ω₁ + (Vₜ₁−Vₜ₂)·2/(5R) ]

where I is the block's moment of inertia and ω its rotational velocity. A consequence of this formulation, confirmed during development by direct calculation: once a block reaches rolling-without-slipping (ω = Vₜ/R), its tangential retention per subsequent rebound scales as √Rₜ rather than Rₜ directly — which is always the larger value for Rₜ < 1. A smoothly rolling block can therefore retain more energy per small rebound than a naive direct-scaling model would predict, while losing more energy during the initial low-spin bounces immediately after release. This is real, intended physics, not an error.

Physical floor on grazing-impact tangential loss

A real validation exercise against a digitized reference case (Section 13) traced a previously unnoticed problem: once a block's velocity becomes nearly tangential to a roughly planar slope (a grazing impact, where the normal velocity component is small relative to the tangential one), the equations above can remove a much larger proportion of tangential speed than is physically possible for that impact. Rigid-body impact mechanics literature (Stronge 1994; reviewed in an academic thesis on rigid-body impact mechanics applied to rockfall trajectory modeling) shows that for a sufficiently grazing impact, the contact slides through the entire impact duration and the tangential loss is bounded by simple Coulomb friction — a small absolute quantity proportional to the (small) normal velocity, not a large proportion of the tangential speed. A physical cap is now applied: the tangential loss from the equations above is capped at μ·(1+Rₙ)·|Vₙ₁| (using a representative rock-on-rock friction coefficient, μ≈0.6) whenever it would otherwise exceed that bound. This activates only for near-grazing impacts; for normal and steep impacts the published equations' own loss is already smaller than this bound, so behaviour there is unchanged. The same cap is applied identically to both the rigid-body and lumped-mass methods (Section 3), since the underlying friction limit doesn't depend on whether rotation is tracked explicitly.

Sustained ground contact: two selectable models

On slopes shallower than 30°, sustained ground contact can be represented by either of two genuinely different, both-published approaches, user-selectable:

  • Short-rebound impacts (default) — a sequence of short-distance rebounds (spaced at the block's own radius, minimum 0.2 m), each using the same rebound equations above. Gravity acts on the block over each hop's travel time before the rebound is applied, which is necessary for the block to gain or sustain speed while rolling; an earlier internal build omitted this and was corrected after the resulting unrealistically short runout was traced and confirmed (Section 13). This treats sustained contact as a sequence of discrete micro-collisions, faithful to the cited source, but a long, fast, low-slope traverse loses energy at every one of potentially thousands of these discrete events — which a digitized validation case (Section 13) showed can understate sustained velocity on long moderate slopes relative to an independent reference.
  • Continuous friction — integrates motion smoothly via Newton's second law with Coulomb friction (net tangential acceleration = g·(sinθ − μcosθ)), with no discrete collision losses at all. Found during validation to better match an independent digitized reference case with documented high sustained velocity on a long moderate slope. Two optional refinements are available within this mode:
    • Variable friction (scarring) — friction increases with accumulated sliding distance on soft ground, μ(s) = μ₀ + Δμ·(1−e−s/L), representing a berm of material building up in front of the block as it plows through; resets whenever the block leaves continuous contact, since the berm belongs to that particular patch of ground, not the block.
    • Viscoplastic ground drag — an additional linear velocity-dependent deceleration (−k·v), distinct from Coulomb friction, representing energy dissipation from viscoplastic ground deformation.
    Both refinements use illustrative preset parameter values (Extra soft/Soft/Medium/Hard), explicitly disclosed in-tool as engineering approximations, not exact calibrated values.

Neither sustained-contact model is more "correct" in general — they are different physical idealizations of the same real contact, and the right choice depends on which behaviour matches a given site's actual rockfall evidence.

Release style

Two release styles are available, both using a velocity aligned to the local slope's tangent plane (via normal projection) rather than an arbitrary fixed direction, so neither produces an unrealistic large first bounce purely from a release/slope mismatch: rolling start begins in continuous contact (falling back automatically to a falling start if the local release-point slope exceeds 30°, since a block can't begin in rolling contact on a cliff face), and free-fall start begins already airborne, representing a block detaching rather than one settled in contact.

4. Rigid-body vs. lumped-mass methods

Both modules compute every trajectory using two methods simultaneously, for direct comparison:

  • Rigid body — the full formulation in Section 3, with rotational coupling and a real moment of inertia derived from the block's actual shape.
  • Lumped mass — the classical simplification in which the tangential velocity is scaled directly by Rₜ with no rotational accounting at all (Vₜ₂ = Rₜ·Vₜ₁), subject to the same grazing-impact friction-bound cap described in Section 3.

Both methods sample the identical block size/density/shape per trajectory (not independently randomized), so any difference reported between them reflects only the physics distinction above, not a confound from comparing different random blocks. An earlier internal build applied the friction-bound cap (Section 3) only to the rigid-body path, leaving the lumped-mass comparison artificially under-protected and exaggerating the gap between the two methods well beyond what the underlying physics predicts; this asymmetry was found and corrected (Section 13), and the methods now agree closely whenever the grazing-impact regime dominates a trajectory, diverging only where the genuine rolling-without-slipping effect described in Section 3 applies.

5. 2D module — methodology

The 2D model takes a hand-entered slope profile (a series of x,z coordinate pairs defining a single representative cross-section) and releases a user-specified number of independent stochastic trajectories from a chosen point along it. Each trajectory samples its own block size (from a lognormal, uniform, or fixed distribution), block shape and density (optionally with run-to-run variability), and surface restitution/roughness values, then integrates the trajectory using a discrete timestep (dt = 0.012 s) switching between free-fall flight and whichever sustained-contact model is selected (Section 3).

Block shape (rigid body)

Six rigid-body forms are supported. Sphere, ellipsoid, rectangular block, and disc use the exact published volume/mass/moment-of-inertia formulas for that geometry. Two additional forms — angular fragment (tetrahedron) and columnar/hexagonal prism — model real shapes (blasted/jointed rock, columnar basalt) not covered by that source; their mass/inertia formulas are reasonable engineering approximations, disclosed as such. All forms are driven by a sampled largest dimension and a user-set aspect ratio (with optional run-to-run variability); the block's effective diameter for ground-impact calculations follows the same convention for all forms: the mean of the two largest dimensions.

Obstacle-induced micro-launches (optional, off by default)

Real terrain has small irregularities (a buried cobble, a root, a surface lip) that a coarse hand-entered profile cannot capture, but which genuinely launch a rolling block into a brief real hop. This is modeled as an occasional small rotation of the block's existing velocity toward the surface normal — capped to a handful of events per trajectory with minimum spacing between them, and confirmed by direct calculation to conserve speed (and therefore kinetic energy) exactly, since it only changes direction, never magnitude. Left off by default after user feedback that the resulting bounces on otherwise-gentle terrain didn't match expectations for a smoothly rolling block.

Crest loss (optional)

Models random material loss at a bench crest — local failure progressively eating into a bench over time. At one or more chosen profile vertices, a direct chord is cut between a point d₁ back along the upslope bench and a point d₂ forward along the downslope face, replacing the original bench-edge corner with a steeper, more direct shortcut — geometrically verified exact by independent hand calculation (Section 13). Sampled fresh per trajectory (each block gets its own randomly-perturbed geometry, matching the real intent of representing bench-to-bench variability rather than one fixed "worst case" profile); the drawn slope line always shows the original unperturbed profile, so a plotted trajectory may show a small visual deviation right at the crest-loss vertex.

Calculation screens

One or more vertical "screens" can be defined at specific x-locations; the tool records each trajectory's first crossing (energy, bounce height, speed, and impact angle) and reports median/P90 statistics per screen — useful for checking conditions at a specific point of interest (e.g. a structure or utility corridor) without needing that point to coincide with a mitigation measure.

Scenario comparison & project save/load

Named scenarios (complete input snapshots) can be saved, then all run and compared side-by-side in a single table — useful for testing several mitigation or material assumptions without manually re-entering inputs each time. A separate save/load function exports/imports all settings as JSON (DEM files for the 3D module must be re-uploaded separately, since raw grid data isn't bundled into the small settings file).

6. 3D module — methodology

The 3D model reads a Digital Elevation Model (Esri ASCII grid, plain CSV grid, or a Wavefront OBJ mesh resampled onto a regular grid via inverse-distance-weighted interpolation) and simulates trajectories across the full terrain surface using the identical rebound physics in 3D vector form. Impact response is computed via vector projection (decomposing velocity using dot products against the local surface normal) rather than a hand-picked 2D basis, which is robust by construction against the sign-convention errors that can arise in simpler implementations.

Source point definition (seeders)

Five ways to define release locations are supported:

  • Automatic — every cell within a top percentage band of the DEM exceeding a minimum slope angle, optionally also requiring local convexity (a discrete Laplacian curvature check), since real rockfall sources cluster on ridges/outcrops rather than uniformly across any steep slope.
  • Point seeder(s) — explicit coordinates for one or more known release locations.
  • Line seeder — points distributed along a defined line (e.g. a mapped scarp or cliff edge).
  • Area seeder — points scattered randomly within a defined rectangle.
  • Plane seeder — points scattered along a defined footprint line, each released from a random height above the local terrain within a specified range, modeling a rock face rather than a point or line source on the ground; always begins in free-fall regardless of the global release-style setting, since there is no surface to be in contact with at the moment of release.

Per-cell spatially-varying surface properties (optional)

Soil type and roughness-zone maps can be uploaded as auxiliary rasters (matching the DEM's exact dimensions and orientation), letting restitution and roughness vary spatially across the terrain instead of using one global value for the whole run. Falls back to the global setting wherever no auxiliary raster is supplied.

Direction and fall-direction deviation

Each trajectory launches in the local steepest-descent direction with a user-set random azimuth jitter. After each rebound, the block's horizontal direction is blended toward the local aspect by an angle sampled from a speed-dependent probability table (smaller deviations are far more likely than large ones; faster blocks deviate somewhat more), capped so it cannot overshoot past the aspect direction itself. The continuous-friction sustained-contact model (Section 3) holds direction along the current horizontal travel vector instead, without this per-step aspect re-blending, since continuous sliding has no discrete "hop" events to apply it to — a disclosed simplification specific to that mode.

Output rasters

Per-cell statistics are aggregated across all simulated trajectories: maximum/mean/P90 translational energy, maximum/P90 rotational and total energy, maximum/mean/P90 bounce height (measured normal to the slope), maximum translational and rotational velocity, passage frequency, reach probability, propagation probability, deposited-block counts (cells where a trajectory genuinely came to rest, distinguished from merely leaving the grid or exhausting the simulation's step budget), number of distinct source cells reaching each cell, and minimum observed energy-line angle per deposit cell.

Spatial hazard indicator

An additional derived overlay synthesises the simulation's two primary spatial outputs — P90 kinetic energy (intensity) and reach probability (a spatial frequency proxy) — into a six-level colour-coded hazard class per cell. The classification uses a 3×3 intensity × frequency matrix:

Energy classLow probabilityMedium probabilityHigh probability
Low (<E₁)LowMedium-lowMedium-high
Medium (E₁–E₂)Medium-lowMedium-highHigh
High (>E₂)Medium-highHighExtreme

Default energy thresholds E₁ = 30 kJ, E₂ = 300 kJ follow the reference values in the Swiss Guidelines (Raetzo et al. 2002) as widely-cited benchmarks; all thresholds are user-adjustable to any jurisdiction's values. Reach-probability thresholds are independently user-adjustable.

Important limitations of the spatial hazard indicator: Reach probability is a spatial frequency proxy, not an event return period. Converting it to a true annual exceedance probability requires site-specific event frequency data (detachment rate per year from geological evidence) that is outside the scope of a trajectory simulator and must be supplied by the practitioner. The energy metric is P90 translational KE from the simulation — a conservative population statistic, not a "design block" derived from proper fragmentation and volume analysis. This overlay is a screening-level tool for identifying relative spatial patterns of hazard intensity and extent; it is not a substitute for a full methodology-compliant hazard assessment conducted by a qualified practitioner using appropriate site data.

Interactive 3D view, animation & click-to-test barriers

A rotatable WebGL terrain view renders the DEM colored by hillshade blended with the currently selected hazard raster, with sample trajectories drawn as real 3D paths; an animation mode interpolates block positions along these paths frame-by-frame, color-coded by peak energy. Clicking anywhere on the rendered terrain runs the barrier adequacy check (Section 9) at that exact location via raycasting from the click point to the terrain mesh, dropping a marker and reporting the result without needing to type coordinates manually. A separate 2D profile analysis tool extracts the elevation profile along a user-defined line through the terrain, together with any simulated trajectories passing within a defined corridor of that line, projected onto the same distance/elevation axes, including a translational/rotational velocity chart along that same line.

8. Forest interaction module

Two genuinely different, both-legitimate forest interaction models are selectable, independently for the 2D and 3D modules.

Discrete tree impacts (default)

Blocks can strike individual trees and lose energy accordingly. The maximum energy a tree can dissipate depends on its trunk diameter and species:

Eᵢᵢᴸᴲ = FE_ratio × 38.7 × DBH²·³¹ (J, DBH in cm)

with FE_ratio = 0.93 for coniferous and 1.59 for broadleaved species. Whether the full amount is actually dissipated depends on the horizontal position of the impact relative to the trunk's central axis, the vertical impact height relative to a theoretical tree height (Hᴳᴷᴷᴸ = 1.22×DBH⁰⁯⁎), and the impact angle relative to the vertical trunk:

Eᴳᴷᴸᴸᴸ = Eᵢᵢᴸᴲ × dEᵛᴸᴷᴷ × dEᵓᵠᵡ × dEᵡᴸᴲᴸᴳ / 1000 (kJ)

3D module: trees are placed explicitly at random positions within the forested zone, drawn from the specified stand density and a normal distribution around the mean trunk diameter, indexed spatially for efficient nearby-tree lookup during simulation. The horizontal miss-distance and impact angle used in the equations above are computed from genuine 3D geometry between the block's path and each tree's real position.

2D module: since a profile has no real lateral dimension, tree strikes are instead modeled probabilistically — an expected-hits-per-step calculation from stand density and swept width, with the horizontal miss-distance sampled from a uniform distribution rather than computed from real tree geometry. This is a disclosed simplification specific to the 2D model; the 3D model's explicit tree placement is the more physically complete implementation.

The published energy-dissipation equations above were checked directly against the original source paper's own worked examples before being implemented (Section 13) and matched closely.

Continuous canopy drag

An alternative model: a steady velocity-dependent damping force applied throughout a defined forest zone and height range, with no individual tree collisions modeled at all — conceptually closer to a continuous viscous medium than discrete obstacles. The exact exponential solution to dv/dt = −(k/m)v is applied over each step's elapsed time:

v₂ = v₁ × e−(k/m)·t

where k is a drag coefficient (kg/s) and m the block mass, active only while the block's height above the surface is within a specified effective height. Density presets (Open/Medium/Dense, spanning roughly 200–800 kg/s) are illustrative engineering approximations, explicitly disclosed as such rather than exact calibrated values; confirmed by direct calculation to produce monotonically increasing velocity loss with density (Section 13). Neither forest model is more "correct" in general — the choice depends on whether individual-tree-scale detail or a simpler zone-averaged effect better suits the available site information.

9. Mitigation design tools

Catch ditch check: given a location, width, and depth, the tool reads the simulated 90th-percentile energy and bounce height at that point, converts energy to an approximate velocity using the mean simulated block mass, and estimates the horizontal launch reach across the ditch via simple projectile kinematics (fall time across the combined ditch depth and bounce height). The reach-to-width ratio is rated on a four-tier scale (Good/Moderate/Limited/No catchment). Translational, rotational, and total kinetic energy at the location are all reported; the reach estimate itself uses translational energy only, since rotation doesn't directly affect ballistic launch distance the way translational velocity does.

Barrier fence check: given a location, energy capacity, physical height (measured normal to the slope), and inclination, the tool reports what percentage of simulated trajectories at that point would exceed either the energy capacity or the effective height — distinguishing a fence that is structurally undersized from one that is simply too short to intercept the bounce trajectory, since these are different failure modes with different fixes. Translational, rotational, and total energy are all reported; the exceedance check itself uses translational energy only, matching how barrier capacity is conventionally specified, with total energy shown for reference.

Inclination

A barrier leaning away from perpendicular to the slope presents a reduced effective profile along the slope-normal direction bounce height is measured against:

Effective height = Physical height × sin(inclination)

At 90° (standing perpendicular to the slope) the full physical height counts, matching default behaviour. This is an explicitly disclosed engineering approximation of the basic geometric effect of leaning a barrier, not an exact published formula, and does not capture the full physics of how a leaning barrier interacts with an oncoming trajectory's actual angle of approach. Verified exact at both reference angles (Section 13).

Optimal placement search

Given a protected-asset location, the tool scans every point upslope and identifies the location with the lowest required capacity (at a chosen design percentile) that still protects the asset — verified directly to find genuine interior minima in non-monotonic energy profiles, not merely the last point before the asset.

Barrier sensitivity analysis

Automatically sweeps a barrier parameter (location, height, capacity, or inclination) over a defined range and plots exceedance and energy against it, optionally as a two-variable grid. Computed instantly from the existing simulation's per-location energy/height distributions, since location/height/capacity/inclination don't change the unobstructed simulated envelope itself, only what counts as "exceeding" it at each setting — verified directly to produce monotonically increasing exceedance as capacity decreases (Section 13). This evaluates the unobstructed simulated envelope against each candidate setting; it does not model how an actual barrier intercepting some rocks would alter what continues downstream, since the barrier checks in this tool are a post-hoc capacity check, not an active obstacle within the simulation.

Illustrative barrier capacity classes

Reference capacity values (100–8,500 kJ) offered as a selection aid are representative points spanning the certified range described in ETAG 027 / EAD 340059-00-0106, not manufacturer-specific data — an actual design should reference a specific certified product's test certificate.

10. Qualitative methods (RHRS / mRHRS)

A separate tool, fundamentally different from the trajectory simulators above: a qualitative scoring system for prioritizing which rock slopes most need mitigation, not simulating any individual trajectory. Two selectable methods, both scoring nine site characteristics on an exponential 3/9/27/81-point scale and summing into a total hazard score:

  • RHRS — the original system developed by Pierson et al. (1990) for the Oregon Department of Transportation. All numeric thresholds verified directly against the official FHWA Participant's Manual (Pierson & Van Vickle, 1993, Report FHWA-SA-93-057) rather than secondary sources.
  • mRHRS — Budetta's (2004) modification, confirmed by direct comparison against the original manual to differ in exactly three categories: geologic character (replaced with Romana's Slope Mass Rating in place of the original's two-case structural/erosion assessment), roadway width (different numeric thresholds), and block volume (a per-boulder volume metric, roughly 5× smaller in scale than the original's per-event released-quantity metric). Climate/water and rockfall history use identical criteria in both versions, confirmed against the primary source.

Geologic character — structural difference between the two methods

This is the largest structural difference. mRHRS takes a single Slope Mass Rating (SMR) value as input. RHRS instead scores two alternative cases independently — Case 1 for slopes where joints/discontinuities dominate (structural condition + rock friction), Case 2 for slopes dominated by differential erosion or oversteepening (erosion features + difference in erosion rates) — and uses whichever case totals higher, matching the official guidance to score both when unsure which applies.

This is a prioritization tool for comparing many sites against each other, not a trajectory simulation — it does not interact with, validate, or get validated by the 2D/3D modules above. SMR itself must be calculated separately from discontinuity mapping; this tool takes it as a direct input, not a full rock-mass classification.

11. Calibration / back-analysis

Given an observed field runout distance, the tool performs a log-scale bisection search over a surface-roughness scale factor (applied to all three roughness classes together, preserving their relative ratios), running a smaller trial batch at each candidate scale and comparing the simulated median stopping location against the observed value. The underlying roughness-to-runout relationship was confirmed monotonic by direct calculation before relying on bisection to search it. This calibrates one relationship (roughness against runout) from one observation; a complete site calibration should cross-check against additional independent evidence (impact crater depths, tree-scar heights) where available.

12. Statistical outputs

Both modules report results as distributions, not single values: median, 90th-percentile, and maximum energy and bounce height along the slope (2D) or per raster cell (3D), since a single Monte Carlo run intentionally samples a population of differently-sized blocks and differently-rough impacts, not one deterministic outcome. When exactly one trajectory is run, the interface relabels these as single-value results rather than population statistics, since percentile language is misleading for a sample size of one — though the underlying run still uses the same stochastic rebound parameters as any other run unless the block-size distribution is explicitly set to "Fixed."

Energy breakdown

Translational kinetic energy, rotational kinetic energy, and their sum (total energy) are tracked and reported separately, per point and aggregated per location/cell. Rotational energy is exactly zero throughout for the lumped-mass method, by definition, since that method does not track rotation at all — confirmed by direct calculation (Section 13), along with confirming total = translational + rotational exactly at every point.

Velocity outputs

Translational and rotational (angular) velocity are both available: as spatial envelope charts along the profile (2D), as a dedicated per-cell raster (3D, mirroring the existing translational-velocity raster), and along a user-defined line through the 3D terrain via the profile-extraction tool. Rotational velocity is, again, zero throughout for the lumped-mass method.

Distribution histograms

In addition to the spatial/profile envelope charts, six distribution histograms are available (bounce height, total/translational/rotational kinetic energy, translational/rotational velocity), each showing the distribution of that trajectory's peak value across the full simulated population — a histogram of per-trajectory peaks, not a spatial envelope. All six are always computed regardless of which is currently displayed, and all six are embedded automatically in PDF report exports.

14. Known limitations

  • Penetration-depth correction to tangential restitution is omitted (Section 3) due to unverifiable source constants — a simplified roughness-only form is used instead.
  • 2D forest interaction uses a probabilistic miss-distance proxy, not real tree geometry (Section 8).
  • Per-cell spatially-varying soil type and roughness are supported in the 3D module via auxiliary raster upload (Section 6); block density, dimensions, and other material properties remain single global values per run.
  • DEM import is limited to ASCII grid, CSV, and OBJ mesh formats; CAD formats (DXF, STL) are not parsed in-tool due to the breadth of entity types involved and the resulting risk of silent misinterpretation — recommended workflow is converting such files to OBJ in GIS/CAD software first, or sharing them directly for manual interpretation.
  • Calibration (Section 11) adjusts one parameter (roughness) against one observation (runout distance); it does not perform a multi-parameter inversion against multiple independent evidence types simultaneously.
  • The interactive 3D view's rendering code (as distinct from its underlying terrain/trajectory data preparation) could not be tested outside an actual browser environment during development.
  • Scarring, viscoplastic ground drag, and continuous canopy drag all use illustrative preset parameter values, explicitly disclosed in-tool as engineering approximations rather than exact calibrated values.
  • Barrier sensitivity analysis (Section 9) evaluates the unobstructed simulated envelope against each candidate barrier setting; it does not model active interception altering downstream trajectories, since barriers in this tool are a post-hoc capacity check rather than an obstacle within the simulation itself.
  • Crest loss (Section 5) perturbs each trajectory's geometry independently; the drawn slope line always shows the original unperturbed profile, so a plotted trajectory may show a small visual deviation right at a crest-loss vertex.
  • The original RHRS's geologic-character Case 1/Case 2 selection (Section 10) requires engineering judgment about which failure mode dominates a given slope; the tool computes both and uses the higher, per official guidance, but does not itself determine which case is geologically appropriate.
  • This tool has not been validated against an instrumented full-scale field rockfall experiment — only against the internal consistency, literature-value, and independent digitized reference-case checks recorded in Section 13.

Two genuinely different, both-published ways to represent sustained contact on gentle terrain. A barrier leaning away from perpendicular to the slope presents a reduced effective profile along the slope-normal direction bounce height is measured against: effective height = physical height × sin(inclination). At 90° (standing straight up from the slope) the full physical height counts, matching the default behaviour before this option existed. This captures the basic geometric effect of leaning a barrier, not the full physics of how a leaning barrier interacts with an oncoming bounce trajectory's actual angle of approach.
Illustrative capacity classes, not manufacturer-specific data. These figures are representative reference points spanning the certified range described in ETAG 027 / EAD 340059-00-0106 (the European testing & certification standard for rockfall protection barriers) — rigid barriers are classified below 100 kJ, deformable/flexible barriers above it, 500 kJ is the most commonly certified capacity in practice, and certified systems span up to >4,500 kJ, with the most extreme tested systems reaching roughly 8,600 kJ. Exact class boundaries vary slightly by manufacturer and certification era — for an actual design, the specific certified product's test certificate governs, not this reference list. Height is measured normal to the slope surface — so a block can clear a vertically-tall-but-slope-perpendicular-short fence on a steep face.
By default the fence is only a post-hoc check (it reads the unobstructed simulated demand but doesn't change anything). Enabling this makes every trajectory in the next "Run simulation" actually interact with the fence as it reaches this location: if the block's energy and height both clear the design values, the fence stops it there and then; if it's too tall to be intercepted, it passes over unaffected; if it's short enough but carries more energy than the rated capacity, it passes through with that capacity's worth of energy removed (a simplified "the barrier absorbs up to its rated capacity, then fails" rule, not a literal deformation model). Everything downstream of the fence — envelope charts, histograms, calculation screens — then reflects this. The "Check fence adequacy" result below is unaffected by this setting either way: it always reports the true incident demand arriving at the fence, which is what you need to size it in the first place.
A counterintuitive but real effect to expect: because trajectories within capacity are stopped outright while only above-capacity ones get a flat deduction and continue, the survivors beyond the fence are a biased subset — weak trajectories are removed entirely, leaving mostly above-capacity ones. For a right-skewed block-size population this can make the energy envelope beyond the fence look higher than you'd naively expect, even though far fewer trajectories get there at all and the fence is doing exactly what it should. Read the interception summary above (stopped/reduced/overtopped counts) alongside the downstream charts, not the downstream energy values in isolation.
